> **项目名称**：音乐推荐系统（Music Recommendation System）  
> **开发时间**：2026年6月  
> **版本**：v2.0（DeepSeek 智能推荐版）

## 一、项目概述

### 1.1 项目背景

随着数字音乐的普及，用户面临着海量音乐内容的选择困难。传统的音乐平台推荐主要依赖协同过滤和基于内容的推荐算法，但这类方法难以理解用户复杂的音乐偏好，推荐结果缺乏解释性。本项目旨在构建一个融合传统推荐算法与**大语言模型（DeepSeek）**的智能音乐推荐平台，为用户提供个性化、可解释的音乐推荐服务。

### 1.2 项目目标

1. 构建一个完整的音乐发现平台，支持歌曲浏览、搜索、收藏、评分、评论等核心功能
2. 实现基于 DeepSeek 大模型的智能推荐引擎，根据用户多维度行为数据生成个性化推荐
3. 提供热门榜单、歌手浏览、播放历史等辅助发现功能
4. 实现歌曲数据多样化展示策略，提升用户体验的自然感与真实感
5. 搭建完整的前后端分离架构，前后端均可独立部署

### 1.3 项目规模

| 指标 | 数值 |
|------|------|
| 歌曲数量 | 1000+ 首 |
| 歌手数量 | 100 位 |
| 音乐流派 | 10 种（Pop/Rock/Folk/HipHop/R&B/Electronic/Jazz/Classical/Metal/Indie） |
| 后端代码文件 | 55 个 Java 源文件 |
| 前端页面 | 13 个页面/视图 |
| 数据库表 | 12 张 |
| Git 提交数 | 20 次 |

### 4.3 核心表设计

| 序号 | 表名 | 说明 | 核心字段 |
|------|------|------|---------|
| 1 | `users` | 用户表 | id, username, password(BCrypt), nickname, email |
| 2 | `artists` | 歌手表 | id, name, original_name, nationality, description, image_url |
| 3 | `songs` | 歌曲表 | id, title, artist_id(FK), album, duration, cover_url, play_count |
| 4 | `favorites` | 收藏表 | id, user_id(FK), song_id(FK), UNIQUE(user_id,song_id) |
| 5 | `ratings` | 评分表 | id, user_id(FK), song_id(FK), score(1-5 CHECK), UNIQUE(user_id,song_id) |
| 6 | `play_history` | 播放历史 | id, user_id(FK), song_id(FK), played_at |
| 7 | `comments` | 评论表 | id, user_id(FK), song_id(FK), content, parent_id(自引用FK, 楼中楼) |
| 8 | `tags` | 标签字典 | id, name, type(genre/mood/artist/era) |
| 9 | `song_tags` | 歌曲标签关联 | id, song_id(FK), tag_id(FK), UNIQUE(song_id,tag_id) |
| 10 | `train_tasks` | 训练任务 | id, status(pending/running/completed/failed), message |
| 11 | `recommendation_results` | 推荐结果 | id, user_id(FK), song_id(FK), score, source(ncf/tag/fusion/deepseek) |
| 12 | `charts` | 榜单数据 | id, song_id(FK), chart_type(daily/weekly/total), hot_score, rank_date |

### 4.4 设计亮点

1. **多语言歌手支持**：`artists` 表同时存储英文名（`name`）和原生名（`original_name`），根据歌手国籍（`nationality`）自动选择合适的显示名称
2. **楼中楼评论**：`comments` 表通过 `parent_id` 自引用实现嵌套回复
3. **推荐来源追踪**：`recommendation_results` 表通过 `source` 字段区分不同推荐引擎（ncf/tag/fusion/deepseek）
4. **评分约束**：`ratings` 表使用 CHECK 约束确保评分在 1-5 之间

## 五、核心功能实现

### 5.1 DeepSeek 大模型推荐引擎（v2.0 核心特性）

#### 5.1.1 架构设计

DeepSeek 推荐引擎是本系统 v2.0 版本的核心创新。它用大语言模型替代了传统的协同过滤算法，通过理解用户的多维度行为数据来生成个性化音乐推荐。

```
用户请求 POST /api/recommendations
         │
         ▼
 RecommendationController
         │
         ▼
 RecommendationService → DeepSeekService.generateRecommend(userId)
         │
         ├── 1. 收集用户行为数据
         │   ├── favoritesMapper.selectUserFavoritesWithSong()    # 收藏列表
         │   ├── ratingsMapper.selectUserRatingsWithSong()        # 评分记录
         │   ├── historyMapper.selectUserHistoryWithCount()       # 播放历史(Top30)
         │   └── tagMapper.selectUserTopTags()                    # 标签偏好(Top10)
         │
         ├── 2. 检查缓存（5分钟 TTL）
         │   └── key = userId:behaviorHash
         │
         ├── 3. 构造 Prompt
         │   ├── 用户偏好标签
         │   ├── 正反馈歌曲（收藏 + 高评分 + 高频播放）
         │   ├── 负反馈歌曲（低评分）
         │   └── 歌曲库摘要（id|歌名|艺人|国籍|标签）
         │
         ├── 4. 调用 DeepSeek API
         │   ├── POST https://api.deepseek.com/v1/chat/completions
         │   ├── model: deepseek-chat
         │   ├── temperature: 0.7
         │   └── max_tokens: 2000
         │
         ├── 5. 解析响应
         │   ├── 提取 JSON 数组 [{"id": 123, "reason": "..."}]
         │   └── 按 ID 批量查询歌曲完整信息
         │
         └── 6. 降级机制（API 不可用时）
             └── 标签匹配 + 热门歌曲补足 + fallback: true
```

#### 5.1.2 Prompt 设计

系统精心设计了三段式 Prompt 结构：

1. **用户偏好**：汇总标签偏好，形成用户画像摘要
2. **反馈分层**：
   - 正反馈：收藏歌曲、高评分（≥4星）、高频播放（≥5次）
   - 负反馈：低评分歌曲（≤2星），用于避免推荐类似歌曲
3. **歌曲库**：完整的歌曲摘要（ID、歌名、艺人、国籍、流派标签）

要求 DeepSeek 以严格的 JSON 格式返回结果：
```json
[{"id": 歌曲ID, "reason": "推荐理由"}, ...]
```

#### 5.1.3 容错设计

| 场景 | 处理方式 |
|------|---------|
| API 超时 | 15 秒超时，降级到标签匹配推荐 |
| API 返回异常 | 捕获异常，自动降级 |
| 响应解析失败 | 正则提取 JSON，兼容 Markdown 代码块包裹 |
| 5 分钟内重复请求 | 返回缓存结果，节省 Token |
| 新用户无行为数据 | DeepSeek 基于歌曲库做多样性推荐 |
| 用户行为变化 | 行为 Hash 变化自动使缓存失效 |

### 5.2 歌曲多样化排序策略

为了消除 AI 生成数据的机械感，使推荐结果看起来像真实音乐平台的天然排序，系统实现了三层多样化策略：

#### 5.2.1 中文歌加权算法

```
排序分 = play_count × (is_chinese ? 1.4 : 1.0)
```

通过 `artist.nationality` 字段判断国籍（中国/台湾/香港/新加坡/马来西亚），对中文歌曲给予 40% 的权重加成，确保推荐流中中文歌占比达到 60-70%。

#### 5.2.2 专辑/歌手打散算法（`SongDiversityUtil.greedyDiverse`）

```java
// 贪心多样化：扫描列表，遇到连续同专辑/同歌手即向后交换
for (i = 0; i < size - 1; i++) {
    if (items[i].album == items[i+1].album) {
        // 向后找第一个不同专辑的元素交换
        for (j = i + 2; j < size; j++) {
            if (items[j].album != items[i].album) {
                swap(items[i+1], items[j]);
                break;
            }
        }
    }
}
```

#### 5.2.3 推荐流交错合并

首页"为你推荐"轮播以 3:1 的比例交错排列中文和外文歌曲，每 3 首中文歌后插入 1 首外语歌，同时确保不产生连续同专辑/同歌手冲突。

### 5.3 华语歌手名称智能显示

系统自动识别歌手国籍，对华语歌手（中国/台湾/香港/新加坡/马来西亚）使用中文名显示，对外语歌手使用英文名显示：

```java
// SongService.applyDisplayName()
if (isChineseNationality(artist.getNationality()) && artist.getOriginalName() != null) {
    song.setArtistName(artist.getOriginalName());  // 显示"周杰伦"
} else {
    song.setArtistName(artist.getName());          // 显示"Jay Chou"
}
```

### 5.4 用户认证系统

- **注册**：用户名唯一性校验 + BCrypt 密码加密
- **登录**：验证用户名密码 → 生成 JWT Token（24小时有效期）→ 返回用户信息
- **认证拦截**：`JwtInterceptor` 拦截需要认证的请求，从 Authorization Header 提取 Bearer Token，解析 userId 并注入到 `request.setAttribute("userId")`
- **前端路由守卫**：`router.beforeEach` 检查 localStorage token，未登录用户访问需要认证的页面时自动跳转到登录页

### 5.5 榜单系统

- 日榜、周榜、总榜三种类型
- 每个榜单 50 首歌
- 热度分计算公式：`hot_score = play_count × 0.4 + fav_count × 0.6`
- 榜单规则：TOP1 必须是中文歌，TOP5 各为不同歌手，中文歌占比 60-70%
- 结果经过专辑/歌手打散处理

### 5.6 评论系统（楼中楼）

支持嵌套回复的评论系统：
- 发表评论时 `parent_id = null` 表示顶层评论
- 回复评论时 `parent_id = 被回复评论的ID`
- `parent_id` 设置 `ON DELETE CASCADE`，删除父评论自动删除所有子回复

## 六、数据处理脚本

### 6.1 数据扩展脚本（`expand_data.py`）

将歌曲数据从约 200 首扩展到 1000+ 首，包含 100 位来自世界各地的歌手：

- **修复已有数据**：为 18 位歌手补充 originalName 和 nationality
- **新增 50 位歌手**：涵盖华语（那英、刘欢、韩红、周深等）、韩国（IU、BTS、BLACKPINK）、日本（YOASOBI、米津玄师）、欧美（Ariana Grande、The Weeknd 等）
- **智能歌曲生成**：根据歌手类型（天王级、摇滚乐队、华语流行等）自动分配合适的流派分布和年代分布
- **预定义精品歌曲**：为周杰伦、林俊杰、陈奕迅、Taylor Swift、Coldplay、Linkin Park、Queen 等 8 组顶级歌手手工定义真实歌曲信息
- **自动生成标签关联**：每首歌自动关联流派标签 + 年代标签 + 艺术家标签

### 6.2 封面填充脚本（`fill_covers.py` + `fill_covers2.py`）

通过 iTunes Search API 为所有歌曲和歌手填充封面图片：

- **分轮次策略**：第一轮快速填充大部分数据，第二轮以更大延迟重试未匹配的歌手
- **搜索策略**：中文歌手用中文名（original_name）搜索，英文歌手用英文名
- **速率控制**：每次请求间隔 0.15-0.6 秒，避免触发 API 限速
- **批量更新**：同一歌手的歌曲共享同一封面 URL

### 6.3 榜单生成脚本（`regenerate_charts.py`）

重新生成自然化的榜单数据：

- 热度值采用递减退化曲线生成（```decay = 1/(1 + rank×0.03)```）
- TOP1 强制为中文歌，TOP5 各不同歌手
- 每歌手最多上榜 3 首
- 添加随机波动模拟真实榜单的自然变化（```noise = random(0.7, 1.3)```）
- 生成后做专辑打散处理

## 八、技术亮点与创新

### 8.1 DeepSeek 大模型推荐

- 与传统推荐算法不同，本系统直接利用大语言模型的语义理解能力来分析用户音乐偏好
- 每首推荐歌曲附带**可解释的推荐理由**（如"因为你喜欢周杰伦的中国风作品，推荐这首..."）
- 设计了三层容错机制：本地缓存 → API 调用 → 标签匹配降级

### 8.2 用户体验优化

- **消除 AI 生成感**：通过多样化排序、长尾分布热度值、随机波动等方式，让数据呈现自然感
- **华语歌手名智能切换**：自动识别国籍，对华语歌手显示中文名，对欧美歌手显示英文名
- **专辑打散算法**：确保同一专辑的歌曲不会连续出现，避免视觉单调
- **前端骨架屏**：推荐加载时展示 skeleton loading 动画，提升感知性能

### 8.3 工程化实践

- **前后端分离**：Spring Boot 3.2 + Vue 3 + Vite 5，支持独立开发和部署
- **分页查询**：所有列表接口支持分页，避免大数据量下的性能问题
- **统一错误处理**：前端 Axios 拦截器统一处理 401 认证失效跳转和错误提示
- **代码生成 + 手工精调**：Python 脚本生成歌曲数据，再手工定义精品歌手的真实歌曲信息
